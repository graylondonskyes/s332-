00 — Read First

This handbook is a control system, not “guidance.” Reps sell a productized outcome (trust conversion) with fixed deliverables and a defensible process. Managers enforce sequence, policy language, and exit criteria. Production enforces intake completeness, direction lock, revision rounds, QA, and file release doctrine.

Commercial Doctrine: 50% deposit to start • 50% before finals Revision Doctrine: 2 rounds • consolidated feedback per round Scope Doctrine: deliverables list is boundary Release Doctrine: finals + sources after final pay clears
Non-negotiable: Any rep who promises “start before deposit,” “finals before final pay,” “unlimited revisions,” or “same-scope discounting” is selling an imaginary product and triggers auto-fail + re-certification.

01 — Operating Doctrine

What is actually being sold, why deals fail in design, and the control laws that prevent chaos.
Outcome = trust conversion Process = certainty Export readiness = deployability Policy breaches = auto-fail
Dense doctrine

1.1 Truth product

We do not sell graphics. We sell trust conversion delivered through a consistent visual system with defined finish lines and controlled iteration. The buyer is purchasing certainty: “I will look established; I will be deployable; I will not be trapped in endless revisions.”

1.2 Root causes of failure

  • Taste-selling: reps talk style and subjectivity instead of outcome and mechanism; buyer can’t confidently decide.
  • Hour-selling: no deliverable boundary, no finish line; scope creep becomes “normal,” timelines slip, payment disputes rise.
  • Price-firsting: quoting before diagnosis makes price feel arbitrary; objections multiply.
  • Revision chaos: unstructured feedback turns into drip edits, shifting directions, and silent resentment.

1.3 Winning mechanism

  • Certainty: fixed deliverables + explicit process = buyer confidence.
  • Control: revision rounds + consolidated feedback = timeline protection.
  • Deployability: export pack discipline = immediate usability (reduces regret and follow-up confusion).

1.4 Control laws

  • Every conversation ends in: deposit OR timeboxed decision checkpoint. No “let me know.”
  • Budget objections are solved by scope downshift, not discounts on the same scope.
  • Policy language is not “suggested.” It is the product boundary: 50/50, rounds, scope, release doctrine.
  • Final deliverables (export pack + sources) release only after final payment clears.

03 — Sales Architecture: Controlled Conversation

Stages with explicit exit criteria; no improvisation; no price-before-diagnosis.
Stages 1–6

Stage flow

  • Stage 1 — Frame + Control (30–60s): earn permission to diagnose. Exit: prospect agrees to focused questions.
  • Stage 2 — Diagnosis (3–7m): capture outcome driver + platform + CTA + deadline + approver; quantify cost of inaction. Exit: rep can state diagnosis paragraph and earn “yes.”
  • Stage 3 — Prescription (60–90s): prescribe one package with “done looks like…” deliverables. Exit: prospect accepts fit logically.
  • Stage 4 — De-risk (60–120s): explain drafts → rounds → export pack; prevent regret. Exit: fear reduces; process understood.
  • Stage 5 — Price + Terms + Close (60–120s): state price + 50/50 + rounds + scope + release doctrine; ask deposit today. Exit: deposit paid OR timeboxed decision set.
  • Stage 6 — Recap + Handoff (immediate): lock scope in writing; convert to production brief. Exit: recap sent + CRM fields complete.
Silent leverage rule: After stating price, stop speaking. The first person to fill silence usually gives away leverage.

05 — Objection Engine: Classification + Trees + Re-close

Every objection gets classified, resolved, and re-closed. No resolution without re-close.
Trees

5.1 Classification model

  • Value uncertainty: fear it won’t deliver; needs mechanism + DoD clarity.
  • Budget constraint: number too high; solved by scope downshift (two-lane close).
  • Authority delay: partner/team approval; solved by decision summary + timebox.
  • Process anxiety: bad past experience; solved by controlled rounds + checkpoints.
  • False stall: “need to think” with no content; solved by classification and timebox.

5.2 Trees (canonical)

“That’s expensive.” Tree
"Fair. Is the hesitation the number itself, or uncertainty it will deliver the upgrade you want?"

IF UNCERTAINTY:
"Let me define done and how revisions work so there’s no regret."
(Deliverables + process certainty)
Re-close: "If we deliver that, are you ready to handle the deposit today?"

IF NUMBER:
"We don’t discount the same scope. We pick the lane that matches budget."
Two-lane close: "Do you want the fast credibility fix, or the full authority system? Which one are we locking today?"
“I need to think.” Tree
"Totally. What part—outcome, timeline, or investment?"

If they answer → resolve that part → re-close:
"If we solve that part, are you ready to handle the deposit today?"

If they can’t answer (stall) →
"If it’s not a yes today, we’ll pause and revisit when timing is better. Before we park it—what would need to be true for this to be a yes?"
Then timebox next step.
“Can you do it cheaper?” Tree
"Price changes when deliverables change. If budget is lower, scope must be smaller."

Present controlled options:
Option A: Downshift to smaller package (fast fix).
Option B: Keep full system.

Re-close:
"Which option are we locking today?"
“Start now, pay later.” Tree
"We can’t start without deposit because it reserves production time. Once deposit clears, we move immediately."

Re-close:
"Want the invoice now so we start today?"

06 — Deal Desk Governance: Authority + Escalation

What reps can offer, what requires approval, and what is forbidden.
No chaos

6.1 Authority levels

  • Rep (no approval): prescribe packages + listed add-ons; downshift scope via tier change; timebox decisions.
  • Team Lead (approval): Rush 72-hour only if rush checklist passes; swap deliverables within same tier only if equal effort and documented.
  • Sales Manager (approval): Rush 48-hour; one close-assist value add up to $150 only if deposit same-day and logged.
  • Exec (approval): any payment term exception; Rush 24-hour; free extra revision rounds; refunds/credits beyond policy; any material scope increases.

6.2 Non-negotiables

  • No start before deposit clears.
  • No final export pack / source files before final payment clears.
  • No unlimited revisions.
  • No discounting the same scope to win.

6.3 Rush eligibility (must be YES)

  • Assets delivered immediately.
  • Approver available same-day.
  • Consolidated feedback per round accepted.
  • Rush fee accepted.
Deal desk law: A concession must be a documented trade. If it is not documented, it does not exist.

07 — Production Governance: Delivery Machine

Intake completeness → direction lock → drafts → revision rounds → QA → final release doctrine.
SOP

7.1 Intake completeness rule

No work begins until intake includes: offer, platform priority, CTA, deadline + tied reason, references (max 3), assets, approver + response window.

7.2 Direction lock rule

Before drafting: outcome driver + style direction + must-include + must-avoid must be documented and internally confirmed. Direction changes after lock are treated as new scope or change order.

7.3 Revision rounds discipline

  • Each revision round requires one consolidated feedback message.
  • Drip edits = production stall. Rep enforces consolidation or issues change order / extra round add-on if behavior persists.
  • “New direction” is not a revision; it is scope change.

7.4 QA + release discipline

Final export pack is prepared and staged. Release occurs only after final payment clears. This prevents disputes and preserves leverage.

08 — Manager QA Governance: Gates + Scorecard

Pass/fail gates, 100-point scorecard, automatic fail triggers, coaching protocol, drill menu.
QA

8.1 Pass/fail gates (all must be YES)

  1. Frame + control achieved
  2. Cost of inaction surfaced
  3. Diagnosis paragraph delivered and confirmed
  4. Single package prescribed first (no menu dump)
  5. De-risk before price
  6. Commercials stated cleanly: 50/50 + rounds + scope + release
  7. Deposit asked today
  8. Written recap sent
  9. Next step timeboxed (if not paid)
  10. CRM fields complete

8.2 100-point scorecard

  • Rapport & Control 15
  • Diagnosis 20
  • Diagnosis Confirm 10
  • Prescription 15
  • De-risk 10
  • Commercials 15
  • Close + Next step 15
  • Pass: 80+

8.3 Automatic fail triggers

  • Start before deposit.
  • Final files before final payment clears.
  • Unlimited revisions promised.
  • Same-scope discounting to close.

8.4 Coaching protocol (mandatory format)

  • Missed gate (what)
  • Correct line (say)
  • Drill (do)
  • Retest time (when)

8.5 Drill menu

  • Rapport stack reps (20/day)
  • Diagnosis sequence reps (10/day)
  • Price + silence reps (10/day)
  • Objection spar (12/day)
  • Deposit close + recap reps (10/day)

11 — Persona Talk Tracks (P1–P6)

Buyer psychology → approved framing → high-leverage questions → value anchors → close strategy.
Persona logic

P1 — New Operator (Credibility Gap)

  • Fear: judged as small/new; low response.
  • Frame: “Not a skill problem—trust signal problem.”
  • Questions: 3-second impression; primary CTA; 30-day cost.
  • Anchor: credibility increases response without changing offer.
  • Close: deposit locks draft date “this week.”

P2 — Busy Service Pro (Consistency Chaos)

  • Fear: paying and still being stuck; time drain.
  • Frame: “You don’t need more effort—need a system.”
  • Questions: bottleneck; weekly cadence; content categories.
  • Anchor: templates buy back time and remove decision fatigue.
  • Close: “next week your content becomes repeatable.”

P3 — Premium/Corporate (Authority + Risk Control)

  • Fear: reputational risk, drift, inconsistent official docs.
  • Frame: “Governed identity system prevents brand drift.”
  • Questions: official contexts; approvals; minimum standard implied.
  • Anchor: governance + documentation = risk reduction.
  • Close: “one-time system protects long-term.”

P4 — Performance (Conversion + Measurement)

  • Fear: “design” is subjective; wants mechanism.
  • Frame: “Not pretty—remove hesitation with trust stack.”
  • Questions: where drop-off occurs; proof assets; top objection.
  • Anchor: variants enable testing; proof reduces friction.
  • Close: wasted spend framing: stop paying for clicks that don’t convert.

P5 — Deadline (Event/Launch Urgency)

  • Fear: public failure; timeline collapse from messy approvals.
  • Frame: “Deadline-controlled; lock deliverables; lock review windows.”
  • Questions: date/time; channels; same-day approvals.
  • Anchor: speed requires discipline and consolidation.
  • Close: deposit reserves production; no deposit = no real draft date.

P6 — Ops/Capacity (Monthly Department)

  • Fear: unpredictable output; chaos disguised as “unlimited.”
  • Frame: “Queue discipline = reliable throughput.”
  • Questions: volume; asset types; request submitter + approver.
  • Anchor: retainer replaces hiring; standards prevent drift.
  • Close: pick tier now; month one deposit starts pipeline.

If price before de-risk: -7

Miss per item: -3 each
Promise finals before final pay: AUTO-FAIL

Must explicitly ask for deposit today.
Weak ask ("let me know"): -10
No ask: -15

No timebox: -8

Recap sent + fields logged.

17 — Deduction Examples (Transcript-Tied)

Exact line-level coaching triggers, what to say instead, and how managers should remediate.

Common deductions

  • Rep says: “We can start whenever.” → Deduction: -10 (Gate 8) + leverage damage. Fix line: “We start once the 50% deposit clears; that reserves production time.”
  • Rep omits: deadline → Deduction: -3 (Gate 2). Fix line: “What’s the deadline and what is it tied to?”
  • Rep omits: cost of inaction → Deduction: -8 (Gate 3). Fix line: “If nothing changes in 30 days, what does that cost?”
  • Rep menu dumps: “We have three packages…” → Deduction: -8 (Gate 5). Fix line: “Based on what you said, the correct fit is [one package].”
  • Rep prices before de-risk: → Deduction: -7 (Gate 6). Fix line: “Process is controlled: drafts → rounds → export pack.” then price.

  • Rep promises finals after deposit: → AUTO-FAIL. Must remediate immediately and re-certify.
